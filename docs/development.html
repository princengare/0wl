development workflow

Build and reload browser-specific WXT targets locally without using a backend, cloud account, or external API.

one command

npm run dev:firefox
npm run dev:chrome
npm run dev:safari

These commands run WXT development builds, launch the selected browser runner, load the extension automatically, and reload after source changes.

WXT writes browser-specific output under .output/ and generated type metadata under .wxt/.

ui style system

0wl uses a terminal-style black-and-white interface across extension pages and docs.

  • Use bundled JetBrains Mono with ss01 and slashed-zero OpenType features.
  • Use bracket checkboxes: [ ] for off and [✓] for on.
  • Underline checkbox rows on hover or keyboard focus.
  • Use custom terminal dropdowns instead of native select controls in the dashboard.
  • Dropdown menus open underneath the field, invert colors on hover/focus/selection, then close after selection.
  • Keep dashboard header tabs on one horizontal line where practical.

manual loop

npm run build:firefox
npm run build:chrome
npm run build:edge
npm run build:opera
npm run build:safari

Firefox manual loading should use .output/firefox-mv3/manifest.json. Chromium-family browsers should load the matching unpacked output folder. Safari assets are generated at .output/safari-mv2/.

safari wrapper

npm run safari:convert
npm run safari:open

Safari testing and distribution require Apple's Safari Web Extension app wrapper. The wrapper is generated under platforms/safari/xcode/. Regenerating it can overwrite generated Xcode resources, so re-check local signing settings.

data safety

  • Use the dedicated .web-ext-profile/ profile for development.
  • Do not commit profile data, IndexedDB snapshots, local exports, or signing secrets.
  • Build output can be recreated; extension storage should be treated as user data.
  • Safari uses a separate Xcode wrapper and requires browser-specific testing.
